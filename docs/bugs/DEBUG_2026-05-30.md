> Document pour Claude Opus. Lit tout avant d'agir.

## 📊 Résumé

Après 10h de debug intensif (par DeepSeek V4 Pro via G-Bot/OpenClaw), **6 bugs majeurs** ont été trouvés et corrigés. Le système produit maintenant des notifications (intention + pipeline compressée fonctionnels), mais un **problème de fiabilité du pipeline d'événements** persiste: les events coulent dans le world store mais n'atteignent plus le buffer d'intention après le 1er cycle.

## 🐛 Bugs trouvés & corrigés aujourd'hui

### 1. MetaObserver crash — `async def` manquant
- **Fichier:** `src/coremind/plugin_host/registry.py:269`
- **Problème:** `def get_all_stats()` → le protocole attend `async def`
- **Erreur:** `TypeError: object list can't be used in 'await' expression`
- **Fix:** Ajouté `async` à la signature (`async def get_all_stats`)
- **Impact:** Le MetaObserver crashait en boucle, 10h+ sans notifications

### 2. Qdrant API — `search` → `query_points` (v1.18.0)
- **Fichier:** `src/coremind/world/snapshot_memory.py:185`
- **Problème:** QdrantClient v1.18.0 n'a plus de méthode `search`
- **Erreur:** `AttributeError: 'QdrantClient' object has no attribute 'search'`
- **Fix:** Remplacé `self._db.search(collection_name=..., query_vector=...)` par `self._db.query_points(collection_name=..., query=...).points`
- **Impact:** La pipeline compressée crashait → fallback vers snapshot JSON complet (32K tokens)

### 3. Pipeline compressée jamais utilisée par IntentionLoop
- **Fichier:** `src/coremind/core/daemon.py` (lignes 340, 351, 470)
- **Problème:** `pipeline_intention = WorldEncodingPipeline(...)` était créé APRÈS `IntentionLoop(pipeline=pipeline_intention)` → `pipeline=None`
- **Fix:** Injecté la pipeline après création: `intention_loop._pipeline = pipeline_intention`
- **Log de confirmation:** `daemon.pipeline_injected_into_intention`
- **Impact:** La pipeline était `daemon.embedding_pipeline_initialised` mais jamais branchée sur l'IntentionLoop → fallback silencieux vers template v1 (snapshot JSON complet)

### 4. Prompt 32K tokens → 5.8K
- **Cause:** Bugs #2 + #3 → fallback vers v1 avec full snapshot JSON
- **Pipeline compressée réparée:** snapshot diff + similar past states + key metrics → **<6000 tokens** au lieu de 32K
- **Log:** `prompt_tokens=5873` (cycle du 30 mai 17:10)
- **Note:** Premier cycle après restart utilise ~22K tokens (pas de diff car pas de snapshot précédent). Cycles suivants: ~5.8K.

### 5. Filtre d'événements trop strict — `confidence >= 0.7`
- **Fichier:** `src/coremind/intention/loop.py:326` (`_is_significant_event`)
- **Problème:** Filtrait les events avec `confidence < 0.7`, mais TOUS les events plugins sont à **confidence = 0.3**
- **Fix:** Changé à `confidence < 0.2`
- **Config associée:** `min_salience = 0.25`, `min_confidence = 0.30` (dans `~/.coremind/config.toml`)
- **⚠️ Le MetaObserver remonte `min_salience` à 0.35** via la policy `lower_salience_when_quiet` — à surveiller

### 6. HA plugin — RpcError avalé silencieusement
- **Fichier:** `plugins/homeassistant/coremind_plugin_homeassistant/main.py` (`_emit_state_change`)
- **Problème:** `except grpc.RpcError: log.error(...); return` — l'erreur était loggée mais jamais propagée → la boucle de reconnexion ne se déclenchait jamais
- **Fix:** Remplacé `return` par `raise` pour propager l'erreur
- **Ajouté:** Heartbeat périodique (30s) qui appelle `stub.HealthCheck()` — si le daemon est muet, force reconnexion
- **Ajouté:** `import contextlib` + `finally: heartbeat_task.cancel()`

## ❌ Ce qui coince encore

### Problème principal: EventBus → Intention buffer

**Symptôme:** Les events coulent dans le world store (380+ events/min) mais n'atteignent pas le buffer d'intention après le premier cycle.

**Cycle typique après restart:**
1. `start-all.sh` → 10/10 plugins, 400+ events/min ✅
2. Grace period (10 min) → event-driven cycle démarre quand même (bypasse la grâce) ⚠️
3. 1er cycle: OK, `candidates=2 produced=1-2`
4. Après le 1er cycle: plus de `threshold_crossed` ni de `observation_buffered`
5. Les events continuent dans le world store (MetaObserver les voit)
6. Mais le `_event_listener` ne les reçoit plus

**Code concerné:**
- `src/coremind/intention/loop.py:_event_listener()` (ligne ~282)
- `src/coremind/core/event_bus.py:EventBus` (ligne 164)
- Subscription via `event_bus.subscribe()` → AsyncGenerator sur asyncio.Queue

**Hypothèses:**
- Le `_event_listener` task crash silencieusement (pas de log d'erreur)
- Ou la subscription EventBus se désynchronise après le 1er `run_cycle()`
- Pas de `bus_overflow` dans les logs (la queue de 1000 ne déborde pas)
- Le `_event_listener` est bloqué dans `await self.run_cycle()` (qui prend ~30s) → pendant ce temps, les events s'accumulent dans la queue

### Piste de fix: timer fallback

Solution pragmatique: passer `event_driven = false` dans la config → cycles toutes les 900s (15 min) via `_scheduler()` au lieu de dépendre du buffer. Moins réactif mais garanti.

```toml
[intention]
event_driven = false  # ← timer-based, pas event-driven
interval_seconds = 900  # cycle toutes les 15 min
```

Ou garder les deux: réparer le `_event_listener` pour qu'il survive ET avoir le timer comme fallback.

                              ▼
                     IntentionLoop.run_cycle()
